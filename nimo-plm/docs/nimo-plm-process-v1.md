# nimo PLM 研发流程体系 V1.0

> 对标 OPPO/华为消费电子研发体系，结合 nimo 智能眼镜业务特点的轻量化 Phase-Gate 流程设计。
> 第一阶段目标：管好 **立项 → EVT → DVT → PVT → MP** 五个阶段。

---

## 一、项目架构：平台 + 产品二级体系

### 1.1 平台项目 (Platform Project)

**定位：** 类似汽车底盘平台，每年迭代 1-2 代，定义核心技术模组。

**平台包含的核心模组：**

| 模组 | 说明 | 位置 |
|------|------|------|
| 波导 (Waveguide) | 光学显示核心 | 镜腿 |
| 光机 (LCoS/MicroLED Engine) | 含 MicroLED 芯片 | 镜腿 |
| 主 FPC 排线 | 连接光机 + 传感器（麦克风/光感/Touch 等） | 镜腿 |
| 主电路板 (Main PCB) | 主控 SoC + 蓝牙 + 存储 | 镜腿 |
| 电池 (Battery) | 供电 | 镜腿 |
| 充电 FPC | 含充电磁吸头 | 镜腿 |
| 镜腿结构件 (Temple Housing) | 左/右镜腿壳体 | 镜腿 |
| 脚套 (Ear Tip) | 耳挂部件 | 镜腿末端 |

**物理架构特征：**
- 所有硬件集中在 **镜腿** 和 **桩头** 中
- 镜框（前框）不走线
- 左右镜腿通过 **蓝牙** 通讯同步数据

### 1.2 产品项目 (Product Project)

**定位：** 基于某一代平台衍生的具体眼镜 SKU，每年可达上百款。

**产品项目特有件：**

| 部件 | 说明 |
|------|------|
| 镜架 (Frame) | 前框，不同时尚设计 |
| 桩头 (Hinge Assembly) | 左/右，连接镜架与镜腿的转轴结构 |

**产品 = 平台模组 + 产品特有件**

产品项目创建时，自动继承所绑定平台的模组 BOM，研发只需补充镜架和桩头等差异件。

### 1.3 项目编码规则

```
平台项目: PLT-{年份}{代数}     例: PLT-2026A, PLT-2026B
产品项目: PRD-{平台代号}-{序号} 例: PRD-26A-001, PRD-26A-002
```

## 二、研发阶段与门控 (Phase-Gate)

### 2.1 阶段总览

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  立项     │───▶│   EVT    │───▶│   DVT    │───▶│   PVT    │───▶│   MP     │
│ Concept  │ G0 │Engineering│ G1 │ Design   │ G2 │Production│ G3 │  Mass    │
│          │    │Validation │    │Validation│    │Validation│    │Production│
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
     │               │               │               │               │
   立项评审        EVT退出评审     DVT退出评审     PVT退出评审     量产放行评审
    (G0)            (G1)            (G2)            (G3)            (G4)
```

### 2.2 各阶段定义

#### 阶段 0：立项 (Concept / Project Initiation)

**目标：** 明确产品定义，评估可行性，获得立项批准。

**关键活动：**
- 市场需求与竞品分析
- 技术可行性评估
- 初步成本估算
- 项目团队组建
- 初步产品规格定义

**交付物：**
| 交付物 | 责任人 | 说明 |
|--------|--------|------|
| 产品需求规格书 (PRD) | 产品经理 | 定义产品功能/性能/外观需求 |
| 可行性评估报告 | 技术负责人 | 关键技术风险评估 |
| 项目计划 | 项目经理 | 里程碑、资源、预算 |
| 概念 BOM (cBOM) | 硬件负责人 | 模组级 BOM，识别关键物料和长交期器件 |

**门控 G0 — 立项评审：**
- 评审方：CEO + 技术VP + 产品VP
- 决策：Go / No-Go / 修改后重审
- 通过后：项目正式立项，分配资源

---

#### 阶段 1：EVT (Engineering Validation Test)

**目标：** 验证工程设计的功能可行性，打通关键技术路径。

**关键活动：**
- 原理图设计 & PCB Layout
- 光学系统设计与仿真
- 结构设计（镜腿/桩头）
- 关键元器件选型与验证
- 功能样机制作（通常手工或小批，5-20台）
- 功能测试（不追求良率和外观）

**交付物：**
| 交付物 | 责任人 | 说明 |
|--------|--------|------|
| EVT BOM (eBOM) | 硬件工程师 | 器件级完整 BOM，标注替代料 |
| 原理图 & PCB 文件 | 硬件工程师 | 经过评审的设计文件 |
| 光学设计报告 | 光学工程师 | 波导/光机参数与仿真结果 |
| 结构设计文件 | 结构工程师 | 3D 模型 + 2D 图纸 |
| EVT 测试报告 | 测试工程师 | 功能验证结果、问题清单 |
| 问题跟踪表 | 项目经理 | Open issues 及解决计划 |

**BOM 管控：**
- EVT BOM 提交后需 **硬件负责人审批**
- 审批通过后状态变为"已发布"
- 修改需提交 ECN（但审批流程相对简单，仅需硬件负责人批准）

**门控 G1 — EVT 退出评审：**
- 评审方：技术负责人 + 各模块 Owner
- 关键判据：功能是否全部打通、关键性能是否达标、Open issue 是否有解决方案
- 决策：进入 DVT / 继续 EVT / 终止项目

---

#### 阶段 2：DVT (Design Validation Test)

**目标：** 验证设计满足产品规格，通过可靠性和认证测试。

**关键活动：**
- 设计优化（基于 EVT 问题修复）
- 开模（结构件模具）
- 小批量试制（50-200台）
- 可靠性测试（跌落、温湿度、盐雾等）
- 安规/认证测试（CE/FCC 等预测试）
- 软件联调

**交付物：**
| 交付物 | 责任人 | 说明 |
|--------|--------|------|
| DVT BOM (dBOM) | 硬件工程师 | 趋于稳定的设计 BOM |
| 模具验收报告 | 结构工程师 | T1/T2 模具验收结果 |
| 可靠性测试报告 | 测试工程师 | 全套可靠性数据 |
| 认证预测试报告 | 认证工程师 | CE/FCC 等预认证结果 |
| 问题跟踪表（更新） | 项目经理 | EVT 遗留 + DVT 新发现 |

**BOM 管控：**
- DVT BOM 提交需 **硬件负责人 + 项目经理** 双重审批
- 变更需 ECN 流程：申请 → 影响分析 → 硬件负责人 + 项目经理审批

**门控 G2 — DVT 退出评审：**
- 评审方：技术负责人 + 产品经理 + 品质负责人
- 关键判据：可靠性是否达标、认证能否通过、BOM 是否稳定
- 决策：进入 PVT / 继续 DVT

---

#### 阶段 3：PVT (Production Validation Test)

**目标：** 验证量产可行性，确保产线能稳定产出合格品。

**关键活动：**
- 试产（500-2000台，使用正式产线）
- 产线调试、SOP 编写
- 良率统计与优化
- 最终认证送测
- 包装设计验证
- 品质标准确认

**交付物：**
| 交付物 | 责任人 | 说明 |
|--------|--------|------|
| PVT BOM (pBOM) | 硬件工程师 | 接近量产 BOM，变更严格管控 |
| 试产报告 | 工厂 IE | 试产数据、良率、节拍 |
| SOP / 作业指导书 | 工艺工程师 | 产线作业标准 |
| 认证证书 | 认证工程师 | 正式认证通过 |
| Golden Sample | 品质 | 标准样品 |

**BOM 管控：**
- PVT BOM 变更需 **完整 ECN 流程**：
  - 申请 → 影响分析（成本/交期/库存）→ 技术负责人 + 项目经理 + 采购/品质审批
- PVT 阶段 BOM 变更自动触发成本影响评估

**门控 G3 — PVT 退出评审：**
- 评审方：CEO + 技术VP + 品质VP + 供应链VP
- 关键判据：良率是否达标、认证是否通过、供应链是否就绪
- 决策：进入 MP / 继续 PVT

---

#### 阶段 4：MP (Mass Production)

**目标：** 正式量产交付。

**关键活动：**
- 正式量产
- 品质监控
- 持续改善

**交付物：**
| 交付物 | 责任人 | 说明 |
|--------|--------|------|
| 量产 BOM (mBOM) | 硬件工程师 | 最终冻结 BOM |
| 量产放行报告 | 品质 | 首批量产品质确认 |

**BOM 管控：**
- 量产 BOM **冻结**，任何变更需 **重大 ECN 流程**：
  - 需评估对在制品/库存/客户的影响
  - 最高级别审批（技术VP + 供应链VP）

## 三、BOM 演进体系

### 3.1 BOM 生命周期

```
概念BOM (cBOM)  →  工程BOM (eBOM)  →  设计BOM (dBOM)  →  试产BOM (pBOM)  →  量产BOM (mBOM)
   立项阶段           EVT 阶段           DVT 阶段           PVT 阶段           MP 阶段
   模组级             器件级             器件级(稳定)        器件级(锁定)        器件级(冻结)
   可自由编辑         审批后发布          ECN 变更           严格 ECN            重大 ECN
```

### 3.2 BOM 不是手动录入，是流程产出

**核心原则：BOM 由研发人员在阶段交付时提交，经审批后成为该阶段的正式 BOM。**

流程：
1. 研发人员在系统中编辑 BOM 草稿
2. 阶段交付时，提交 BOM 审批
3. 审批通过 → BOM 状态变为"已发布"，版本号递增
4. 已发布的 BOM 不可直接修改
5. 要修改已发布 BOM → 必须发起 ECN 变更流程
6. ECN 审批通过 → 系统自动更新 BOM 并生成新版本

### 3.3 平台 BOM 继承

产品项目创建时：
1. 选择绑定的平台项目
2. 系统自动复制平台当前阶段的 BOM 作为产品 BOM 的基础
3. 研发补充产品特有件（镜架、桩头等）
4. 平台 BOM 变更时，系统通知所有关联产品项目评估影响

## 四、ECN 变更管控

### 4.1 变更严格度随阶段递增

| 阶段 | 变更审批 | 流程 |
|------|----------|------|
| 立项 | 无需 ECN | 概念 BOM 可自由编辑 |
| EVT | 轻量 ECN | 硬件负责人审批即可 |
| DVT | 标准 ECN | 硬件负责人 + 项目经理 |
| PVT | 严格 ECN | 技术负责人 + PM + 采购/品质 |
| MP | 重大 ECN | 技术VP + 供应链VP |

### 4.2 ECN 流程

```
发起变更申请
    │
    ▼
填写变更内容（改什么、为什么改）
    │
    ▼
影响分析（系统自动关联：受影响的产品、库存、成本变化）
    │
    ▼
审批（按阶段对应的审批链）
    │
    ▼
实施（系统自动更新 BOM，生成新版本，记录变更历史）
    │
    ▼
验证关闭
```

### 4.3 平台变更的级联影响

平台 BOM 变更时，系统自动：
1. 识别所有使用该平台的产品项目
2. 生成影响评估报告
3. 通知各产品项目负责人
4. 各产品项目决定是否同步变更

| CEO/VP | 立项审批、重大决策 | G0/G3/G4 评审、重大 ECN 审批 |

## 六、系统重构方向

基于以上流程，PLM 系统需要重构的核心模块：

### 6.1 项目管理（重构）
- 支持 平台项目 / 产品项目 两种类型
- 产品项目绑定平台项目
- 阶段门控：每个阶段有明确的交付物 checklist
- 阶段推进需所有必要交付物提交完成 + 评审通过

### 6.2 BOM 管理（重构）
- BOM 按阶段版本管理（cBOM/eBOM/dBOM/pBOM/mBOM）
- 研发提交 → 审批 → 发布 的流程驱动
- 已发布 BOM 只读，修改走 ECN
- 平台 BOM 继承机制
- BOM 对比（版本间 diff）

### 6.3 ECN 变更（增强）
- 变更严格度与项目阶段关联
- 自动影响分析（关联产品、成本变化）
- 审批链按阶段动态生成
- 审批通过后自动更新 BOM

### 6.4 交付物管理（新增）
- 每个阶段定义必要交付物模板
- 交付物提交与审批
- 阶段门控自动检查交付物完成度

### 6.5 评审管理（新增）
- 阶段评审（Phase Gate Review）
- 评审结论记录（Go/No-Go/条件通过）
- 评审遗留问题跟踪
