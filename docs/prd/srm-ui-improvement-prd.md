# SRM 供应链管理系统 UI 改善 PRD

> 版本: v1.0 | 日期: 2026-03-29 | 编制: Claude (基于浏览器全页面审计)
> 范围: ACP SRM 模块全部 12 个页面的定制化 UI 改造

---

## 一、现状分析

### 1.1 当前架构

SRM 模块采用 ACP SDK 的**实体驱动自动 CRUD** 架构：后端定义 `EntityDef`（字段、列表列、筛选器、关联），前端根据实体定义自动渲染标准的列表/表单/详情页面。

**优势**: 开发速度极快，新增实体即可获得完整 CRUD。
**问题**: 所有页面都是相同的"表格 + 新建弹窗"模板，无业务语义、无上下文关联、无数据可视化，与专业 SRM 系统差距巨大。

### 1.2 审计发现的关键问题（按严重性排序）

#### P0 — 阻断性问题（影响基本可用性）

| # | 问题 | 影响范围 | 根因 |
|---|------|---------|------|
| 1 | **UUID 暴露** — 供应商、PO、检验单等页面大量显示原始 UUID 而非人类可读的名称/编码 | 全部关联字段 | `relation` 类型字段 `RefDisplay` 配置正确，但前端渲染层未做关联解析 |
| 2 | **模板变量污染** — 数据中存在 `{{steps.step_create_supplier.output.supplier_id}}` 等原始模板表达式 | 流程创建的记录 | 流程 mock 调试产生的脏数据，或表达式未正确解析 |
| 3 | **详情页标题显示 UUID** — 进入详情页后，页面标题/面包屑显示的是 UUID 而非编码或名称 | 所有实体详情页 | 通用详情页组件未使用 `RefDisplay` 字段渲染标题 |

#### P1 — 严重体验问题

| # | 问题 | 影响范围 |
|---|------|---------|
| 4 | **枚举值未本地化** — 列表中显示 `electronic`、`structural`、`pending` 等英文原始值，而非对应的中文标签 | 所有 select 类型字段 |
| 5 | **子表无法展开** — PR 的物料明细、PO 的订单明细、检验的检验项等 `has_many` 关联在列表中不可见，需进入详情才能看到 | PR/PO/检验/结算 |
| 6 | **关键日期字段为空** — 需求日期、预计交期、检验日期等核心日期字段大量为空 | PR/PO/检验 |
| 7 | **L1/L2 资质页列数过多** — 品类资质 18 列、物料认证 21 列，表格横向溢出严重 | 品类资质/物料认证 |
| 8 | **无 Dashboard 总览** — 进入 SRM 模块没有全局概览，直接看到第一个子页面 | SRM 模块入口 |

#### P2 — 功能缺失

| # | 问题 | 专业 SRM 对标 |
|---|------|-------------|
| 9 | 无看板视图（已定义但未使用） | SAP Ariba / Jaggaer 均有供应商看板 |
| 10 | 无数据导入/导出 | 批量导入供应商/物料是基本需求 |
| 11 | 8D 纠正措施过于简化 | 缺少 D1~D8 标准步骤跟踪 |
| 12 | 检验无明细项 | IQC 需要逐项记录检验结果 |
| 13 | 结算无付款跟踪 | 缺少发票/付款/对账明细 |
| 14 | 库存无出入库流水 | 只有当前数量，缺少变动历史 |
| 15 | 供应商无证件/附件管理 | 营业执照、ISO 证书等 |

## 二、改善目标

### 2.1 设计原则

1. **业务语义化** — 每个页面的布局应反映该业务环节的核心关注点，而非千篇一律的表格
2. **上下文关联** — 从供应商能直达其 PO/检验/评估，从 PO 能看到关联 PR 和收货记录
3. **数据可视化** — 关键指标用图表/进度条/仪表盘呈现，而非纯数字
4. **渐进式复杂度** — 列表页展示核心信息，详情页展开完整上下文

### 2.2 对标参考

| 功能域 | 参考系统 | 参考点 |
|--------|---------|-------|
| 供应商管理 | SAP Ariba SLP | 供应商 360° 视图、分级看板 |
| 采购订单 | Oracle Procurement Cloud | PO 生命周期时间线、收货进度 |
| 来料检验 | QMS (SPC) 系统 | 检验明细表、合格率趋势 |
| 供应商评估 | Coupa BSM | 雷达图评分、历史趋势 |
| 8D 改进 | SAP QM | D1-D8 步骤卡片、限期跟踪 |

### 3.2 数据源

| 指标卡 | SQL/API | 刷新策略 |
|--------|---------|---------|
| 供应商数 | `COUNT(*) FROM srm_suppliers WHERE status='active'` | 页面加载 |
| 待处理PR | `COUNT(*) FROM srm_purchase_requests WHERE status IN ('draft','pending')` | 页面加载 |
| 进行中PO | `COUNT(*) FROM srm_purchase_orders WHERE status IN ('approved','sent','partial')` | 页面加载 |
| 待检验 | `COUNT(*) FROM srm_inspections WHERE status='pending'` | 页面加载 |
| 本月结算 | `SUM(final_amount) FROM srm_settlements WHERE period_end >= 本月1日` | 页面加载 |

### 3.3 实现方式

SDK `ViewDef` 新增 `type: "dashboard"`，组件定义通过 `DashboardConfig` 配置项传入。或在 SRM 模块中注册自定义 Component（`sdk.ComponentDef`），前端根据 Component 类型渲染对应的 Dashboard 组件。

## 四、采购全链路看板（核心视图）

> **设计理念**: 参考 Oracle Project Procurement Command Center 和 BOMIST，采用**项目驱动**的采购看板。硬件制造业（nimo 智能眼镜）的采购天然围绕项目/BOM 组织——每轮 EVT/DVT/PVT 都有自己的 BOM，采购团队按项目跟踪所有物料的流转进度。
>
> **行业对标**: 间接采购类 SRM（Coupa/Precoro）采用扁平看板+筛选；但硬件制造类 SRM（Oracle/BOMIST）一致采用项目驱动看板。我们的场景属于后者。

### 4.1 双模式设计：项目看板（主） + 全局看板（辅）

| 模式 | 入口 | 数据范围 | 使用场景 |
|------|------|---------|---------|
| **项目看板** | 从采购项目详情进入 | 该项目下所有 PR Item | 日常工作：跟踪某轮 BOM 采购进度 |
| **全局看板** | SRM 侧边栏顶级入口 | 所有进行中的 PR Item | 管理视角：跨项目看全局瓶颈、逾期 |

两种模式共享同一套看板组件，区别仅在于数据筛选范围。

### 4.4 看板列定义

| 看板列 | PR Item status | PO 状态 | 检验状态 | 说明 |
|--------|---------------|---------|---------|------|
| **需求** | `pending` | — | — | PR 刚创建，物料待处理 |
| **审批** | PR status = `pending` | — | — | PR 提交待审批 |
| **已下单** | `ordered` | `draft`~`sent` | — | PO 已生成（量产）或打样通过后下单 |
| **已发货** | `shipped` | `sent` | — | 供应商已发货在途 |
| **已收货** | `received` | `received`/`partial` | — | 到货确认 |
| **IQC** | `inspecting` | — | `pending`/`inspecting` | 来料检验中 |
| **入库** | `passed` | `completed` | `completed`(pass) | 检验合格，已入库 |
| **完成** | `completed` | — | — | 全流程完成 |

> 相比之前的 9 列精简为 **8 列**，去掉"已批准"（PR审批通过后会立即生成PO，这个状态极短暂不需要独立列）。

**打样物料扩展列:**

打样物料在"审批"和"已下单"之间增加 3 个专属列：

```
需求 → 审批 → [寻源中] → [打样中] → [打样验证] → 已下单 → 已发货 → 已收货 → IQC → 入库 → 完成
```

| 看板列（打样专属） | PR Item status | 说明 |
|-------------------|---------------|------|
| **寻源中** | `sourcing` | 正在寻找供应商 |
| **打样中** | `sampling` | 样品制作/寄送中 |
| **打样验证** | `verifying` | 样品到达，正在验证 |

打样通过后卡片进入"已下单"列，后续与量产一致。

### 4.5 卡片设计

```
┌──────────────────┐
│ CNC外壳           │  ← 物料名称（大号加粗）
│ STR-001 · 100pcs │  ← 编码 · 数量
│ 深圳精密结构件     │  ← 供应商名称（无供应商显示"待寻源"）
│ ¥4,500           │  ← 金额
│ PO-0007 · 交期4/15│ ← 关联PO · 交期（有PO后显示）
│ 量产 🟢           │  ← 采购类型标签
└──────────────────┘
```

**卡片颜色规则:**
- 🟢 绿色左边框 = 量产采购（已认证供应商）
- 🔵 蓝色左边框 = 打样采购（待认证/寻源）
- 🔴 红色左边框 = 逾期（超过预计交期）
- 🟡 黄色左边框 = 即将到期（7 天内）

### 4.6 卡片交互

| 操作 | 触发方式 | 效果 |
|------|---------|------|
| 查看详情 | 点击卡片 | 右侧抽屉：物料信息 + 全链路时间线 |
| 拖拽移动 | 拖拽到下一列 | 触发状态变更 API |
| 快捷操作 | 卡片右上角 `⋯` | 上下文菜单：审批/下单/收货/检验/8D |
| 筛选 | 顶栏筛选器 | 按项目/供应商/品类/优先级/交期过滤 |

**拖拽规则:**
- 只能向右（不能回退状态）
- 前置条件不满足时弹提示（如"已下单"需先有 PO）
- 拖拽到"已收货"自动弹出收货确认表单

## 六、通用 UI 改进

### 6.1 关联字段显示修复 (P0)

**问题**: `relation` 类型字段在列表和详情页显示 UUID 而非名称。

**方案**: 前端通用列表组件在渲染 `relation` 类型字段时:
1. 根据 `RefEntity` 和 `RefModule` 确定目标实体
2. 根据 `RefDisplay` 字段名确定显示字段
3. 批量查询关联记录（列表页用 `IN` 查询避免 N+1）
4. 渲染为可点击链接，点击跳转到关联记录详情

**预计改动**: 前端通用组件层，一次修复所有模块的关联字段显示。

### 6.2 枚举值本地化 (P1)

**问题**: `select` 类型字段显示原始值（`electronic`）而非标签（`电子`）。

**方案**: 前端通用组件在渲染 `select` 类型时，使用 `Options` 中的 `Label` 替代 `Value` 显示，同时应用对应的 `Color` 配色。

**预计改动**: 前端通用组件层。

### 6.3 详情页标题 (P0)

**问题**: 详情页面包屑和标题显示 UUID。

**方案**: 详情页组件根据实体定义的首个非 ID、非隐藏字段（通常是 `code` 或 `name`）作为页面标题。或通过实体定义新增 `TitleField` 配置项。

### 6.4 子表内联展示 (P1)

**问题**: `has_many` 关联在详情页才能看到，列表页无提示。

**方案**:
1. 列表页的关联字段列显示子记录数量（如 `3 项`）
2. 支持行展开（expandable row）显示子表
3. 详情页的子表使用 Tab 组织

### 6.5 日期/时间字段增强

1. 相对时间显示（`3 天前` / `2 小时后`）
2. 逾期/即将到期用红色/橙色高亮
3. 空日期字段显示 `—` 而非空白

### 6.6 批量操作

1. 列表支持多选
2. 支持批量状态变更（如批量审批 PR）
3. 支持 Excel 导入/导出（供应商批量导入尤为重要）

## 七、实现方案

### 7.1 技术路线选择

**方案 A: 扩展 SDK ComponentDef（推荐）**

在 `sdk.ComponentDef` 中定义定制化组件，前端根据组件类型加载对应的 React 组件。

```go
// 在 SRM 模块的 Components() 方法中返回
func (m *SRMModule) Components() []sdk.ComponentDef {
    return []sdk.ComponentDef{
        {
            Name: "srm_dashboard",
            Type: "dashboard",
            Config: map[string]any{...},
        },
        {
            Name: "supplier_360",
            Type: "entity_detail_custom",
            Entity: "suppliers",
            Config: map[string]any{...},
        },
    }
}
```

**优势**: 不破坏通用 CRUD 架构，定制组件与通用组件共存。
**工作量**: 中等。需要前端开发每种定制组件。

**方案 B: 纯前端覆盖**

在 ACP 前端为 SRM 的每个实体注册自定义渲染器，覆盖默认的列表/详情组件。

**优势**: 完全自由的 UI 控制。
**劣势**: 与通用 CRUD 脱钩，后续实体变更需要同步修改前端。

**推荐**: **先做 P0 通用修复（关联字段/枚举/标题），再用方案 A 逐步增加定制组件。**

### 7.2 实施优先级

| 阶段 | 内容 | 预期效果 |
|------|------|---------|
| **Phase 1** | P0 通用修复（关联字段、枚举本地化、标题）+ **采购全链路看板** | 基本可用 + 核心工作台 |
| **Phase 2** | 供应商 360° 视图 + 卡片详情抽屉 + 打样看板扩展 | 全链路可视化 |
| **Phase 3** | SRM Dashboard + 供应商看板 | 管理层概览 |
| **Phase 4** | 检验明细/8D 步骤/库存流水/结算跟踪 | 专业功能深度 |
| **Phase 5** | L1/L2 认证流程视图 + 导入导出 | 完整 SRM 体验 |

> **Phase 1 重点**: 采购看板是 SRM 日常工作的主界面，优先级最高。P0 通用修复也在 Phase 1 完成，因为看板卡片需要正确显示供应商名称（关联字段解析）和中文状态标签（枚举本地化）。

### 7.3 数据模型调整

| 调整项 | 涉及表 | 说明 |
|--------|-------|------|
| 8D 步骤字段 | `srm_corrective_actions` | 增加 `d1_team` ~ `d8_closure` 文本字段或 `steps_json` JSON 字段 |
| 实体定义 TitleField | SDK EntityDef | 新增 `TitleField string` 配置项 |
| L1/L2 ListColumns 精简 | entities.go | 精简为 7-8 列 |
