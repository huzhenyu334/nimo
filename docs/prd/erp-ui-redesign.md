> 版本: v1.0 | 日期: 2026-04-02 | 状态: Draft
> 设计理念: 数据来找你，而不是你去找数据
> 对标: Superhuman(Triage) × Linear(键盘) × Arc(Space) × Stripe(数据美学)

## 一、设计哲学

### 传统 ERP 的根本问题

传统 ERP 是**数据库浏览器** — 用户在几十个菜单、几百个表格里搜索自己需要的数据。这个模型有三个致命缺陷：

1. **方向错了** — 用��适应系统，而不是系统适应用户
2. **无优先级** �� 100 个待处理项平铺，不知道先做哪个
3. **断裂的上下文** — 看订单时看不到库存、看不到客户信用、看不到发货状态

### 我们的答案

```
传统:  用户 → 导航菜单 → 找到数据 → 决定做什么 → 操作
我们:  系统 �� 推送待办 ��� 用户判断 → 一键操作 → 下一个
```

**三个核心原则：**

1. **Triage First** — 系统主动告诉你"现在该做什么"，按紧急度排序，逐条处理
2. **Space Isolation** — 不同角色看到完全不同的界面，只看到与自己相关的一切
3. **Keyboard Native** — 全键盘可操作，Cmd+K 命令面板贯穿所有场景

## 二、信息架构

### 2.1 Space 体系

Space 是角色级的工作空间，每个 Space 有独立的 Triage 队列、视图和操作集。

| Space | 角色 | 核心问题 | 主要操作 |
|-------|------|---------|---------|
| 🛒 销售 | 销售人员/销售主管 | 客户要什么？什么时候交？收到钱了吗？ | 报价→订单→发货→收款 |
| 📦 仓储 | 仓管员/物流 | 物在哪？今天入什么出什么？ | 入库/出库/盘点/发货 |
| 🏭 生产 | 生产计划/车间 | 做什么？缺什么料？做了多少？ | MRP→工单→领料→报工 |
| 💼 财务 | 会计/财务主管 | 赚了多少？欠多少？花了多少？ | 凭证→对账→报表 |
| 📊 管理 | CEO/运营总监 | 整体业务健不健康？ | Command Center 全局仪表盘 |

用户登录后进入自己的默认 Space，可通过左上角切换。

## 三、Triage 系统设计

### 3.1 Triage 是什么

Triage 是一个**智能待办队列**，系统根据业务规则自动生成待处理项，按紧急度排序。用户逐条处理，处理完自动显示下一条。

### 3.2 Triage 项的数据结构

```typescript
interface TriageItem {
  id: string;
  type: string;           // order_pending, low_stock, overdue_invoice, oqc_needed, ...
  priority: 'critical' | 'high' | 'medium' | 'low';
  space: string;          // sales, warehouse, production, finance, quality
  
  // 显示
  icon: string;
  title: string;          // "SO-0045 · 华为光学 · ¥1.13M"
  subtitle: string;       // "已确认 3 天，BOM 物料充足，可以安排生产"
  
  // 关联实体
  entity_type: string;    // sales_orders, inventory, work_orders, ...
  entity_id: string;
  
  // 操作
  primary_action: Action; // 主操作按钮
  secondary_actions: Action[];
  
  // 元数据
  created_at: number;
  due_at?: number;
  assignee_id?: string;
}

interface Action {
  label: string;          // "安排生产"
  command: string;        // "create_work_order"
  params: Record<string, any>;
  shortcut?: string;      // "Enter"
  color?: string;         // "blue" | "green" | "orange"
}
```

### 3.3 Triage 项生成规则

系统根据业务状态自动生成 Triage 项，每次打开 Triage 实时计算：

**销售 Space:**

| 规则 | 优先级 | 标题 | 主操作 |
|------|--------|------|--------|
| 订单状态=draft 且创建>24h | high | 待确认订�� | 确认订单 |
| 订单状态=confirmed 且物料充足 | high | 可安排生产 | 创建工单 |
| 订单状态=ready 且未创建发货单 | critical | 待发货 | 创建发货单 |
| 发票逾期>7天 | critical | 逾期应收 | 发送催款 |
| 发票逾期<=7天 | medium | 即将到期 | 查看详情 |
| 报价有效期<=3天 | high | ��价即将过期 | 联系客户 |
| 退货申请待审批 | high | 退货待审批 | 审批退货 |

**仓储 Space:**

| 规则 | 优先级 | 标题 | 主操作 |
|------|--------|------|--------|
| 物料库存<安全库存 | critical | 低库存预警 | 创建采购申请 |
| PO 状态=shipped（在途到货） | high | 待入�� | 扫码入库 |

| 发货单状态=picking | high | 待拣货发货 | 确认发货 |
| 盘点差异>阈值 | medium | 盘点异常 | 调整库存 |

**生产 Space:**

| 规则 | 优先级 | 标题 | 主操作 |
|------|--------|------|--------|
| MRP 有未确认建议 | high | MRP 建议待确认 | 确认建议 |

| 工单逾期 | critical | 工单逾期 | 调整计划 |

**财务 Space:**

| 规则 | 优先级 | 标题 | 主操作 |
|------|--------|------|--------|
| 有未过账凭证 | medium | 待过账凭证 | 过账 |
| 月末未结转 | critical | 期末结转 | 执行结转 |
| 应收逾期>30天 | high | 应收催收 | 生成催款函 |
| 收款未核销 | medium | 待核销 | 自动匹配 |
| 供应商发票待确认 | high | 待确认应付 | 确认入账 |

**键盘操作**：

| 快捷键 | 操作 |
|--------|------|
| `↑` / `↓` 或 `j` / `k` | 上下移动焦点 |
| `Enter` | 执行主操作（蓝色按钮） |
| `→` 或 `l` | 展开查看详情 |
| `←` 或 `h` | 返回列表 |
| `d` | 延后（Snooze）— 选择 1h/4h/明天/下周 |
| `x` | 标记完成/忽略 |
| `⌘K` | 打开命令面板 |
| `⌘N` | 新建（根据当前 Space 自动选择类型）|
| `1-5` | 切换优先级筛选 |

**单条处理模式**（类 Superhuman）：处理完一条自动滑入下一条，给用户"清空收件箱"的成就感。

func (m *ERPModule) triageRoute(c *gin.Context) {
    space := c.Query("space")
    items := m.computeTriageItems(space)
    // 按优先级排序
    sort.Slice(items, func(i, j int) bool {
        return priorityOrder[items[i].Priority] < priorityOrder[items[j].Priority]
    })
    c.JSON(200, gin.H{"items": items, "counts": countByPriority(items)})
}

func (m *ERPModule) computeTriageItems(space string) []TriageItem {
    var items []TriageItem
    db := m.Deps.DB
    
    if space == "" || space == "sales" {
        items = append(items, m.salesTriageItems(db)...)
    }
    if space == "" || space == "warehouse" {
        items = append(items, m.warehouseTriageItems(db)...)
    }
    // ... production, finance, quality
    
    return items
}

### 6.3 实时更新

使用 SSE (Server-Sent Events) 推送 Triage 变更：

```
GET /api/m/erp/triage/stream
  → SSE: data: {"type":"new","item":{...}}
  → SSE: data: {"type":"resolved","item_id":"xxx"}
  → SSE: data: {"type":"updated","item":{...}}
```

前端通过 EventSource 监听，自动更新 Triage 列表和计数。
