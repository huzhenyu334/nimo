> 版本: v1.0 | 日期: 2026-04-02

## 一、设计第一性原理

### 1.1 企业软件为什么体验差？

不是设计师的问题，是**设计目标错了**：

| 传统企业软件目标 | 我们的目标 |
|----------------|-----------|
| 展示所有功能 | 只展示**此刻需要的** |
| 让用户能找到数据 | **数据主动找到用户** |
| 提供完整的操作 | 提供**最短路径的操作** |
| ��息完备 | **信息分层**：概要→细节→底层 |
| 功能覆盖 | **决策支持**：不只给数据，给建议 |

### 1.2 五条铁律

**1. 3 秒规则** — 打开任何页面，3 秒内必须知道"我该做什么"

**2. 0 跳转操作** — 80% 的操作在当前页面完成，不跳页

**3. 渐进式信息披��** — 默认展示 20% 关键信息，需要时再展开 80%

**4. 键盘 > 鼠标 > 触控** — 高频操作全键盘可达，鼠标是备选

**5. 每个像素都有意义** — 不为装饰添加元素，每个 UI 元素要么传递信息，要么支持操作

**Status Flow 设计要点**（借鉴 Linear 的 Issue 状态条）：
- 圆点连线，已完成的实心 `●`，当前高亮脉动，未来空心 `○`
- 当前状态到下一状态之间有渐变色动画
- 下一步操作按钮直接嵌在状态条右侧
- 不需要找菜单 — 看到状态条就知道下一步该做什么

**Smart Buttons 设计**（借鉴 Odoo，但更精致）：
- 卡片式按钮，显示实体类型图标 + 数量
- 数量 > 0 时卡片亮色背景，可点击
- 数量 = 0 时暗色，hover 显示"点击创建"
- 点击展开**内联关联列表**（不跳页），类似 Notion 的 linked database

**库存状态指示器**（产品明细行内）：
- 🟢 充足 (库存 > 订单量 × 1.5)
- 🟡 紧张 (库存 < 订单量 × 1.2)
- 🔴 不足 (库存 < 订单量)
- hover 显示具体数字：`现有 1200 / 预留 300 / 可用 900`

**表格设计要点：**

- 金额列：主金额 + 已收款子信息
- 状态列：圆点色标 + 文字 + 可选进度条

**b) Hover 预览卡**（借鉴 Figma 的 Inspect 面板）

- 时间线：按交期/创建日期在时间轴上展示

- 拖拽列头调整顺序和宽度
- 右键列头 → 固定/隐藏/排序/筛选
- 用户列配置自动保存到 localStorage

## 五、动效规范

### 5.1 动效原则

**设计标杆**: Apple Human Interface Guidelines + Framer Motion

1. **有目的** — 每个动���都服务于功能（引导注意力 / 提供反馈 / 建立空间关系）
2. **快** — 大多数动��� 100-200ms，绝不超过 300ms
3. **自然** — 使用 spring 物理���擎而非线性 easing
4. **不阻塞** — 动效期间用户可以操作（不锁定 UI）

### 5.2 动效清单

| 场景 | 动效 | 时长 | 缓动 |
|------|------|------|------|

| 下一张卡片补位 | slide-up | 150ms | spring(300, 20) |
| 页面切换 | cross-fade | 100ms | ease |
| Modal 打开 | scale(0.95→1) + fade-in | 150ms | spring(400, 25) |
| Modal 关闭 | scale(1→0.95) + fade-out | 100ms | ease-in |
| Toast 通知进入 | slide-left + fade-in | 200ms | spring(300, 20) |
| Toast 消失 | fade-out | 150ms | ease-in |
| Hover Card 出现 | scale(0.97→1) + fade-in | 120ms | ease-out |
| 数字��化 | count-up/down 动��� | 300ms | ease-out |

## 八、性能目标

| 指标 | 目标 | 说明 |
|------|------|------|
| FCP (First Contentful Paint) | < 1.0s | 骨架屏快速出现 |
| TTI (Time to Interactive) | < 2.0s | Triage 卡片可操作 |
| 搜索响应时间 | < 200ms | Cmd+K 搜索结果 |
| 页面切换 | < 100ms | Space/视图切换 |
| 列表滚动 | 60fps | 虚拟滚动，1万行不卡顿 |
| Triage 刷新 | < 500ms | 实时计算待办项 |
| 动效帧率 | 60fps | 所有动效不掉帧 |

**技术手段：**
- React.lazy + Suspense 按 Space 代码分割
- 虚拟滚动（react-window）处理大列表
- SSE 实时推送避免轮询
- Service Worker 缓存静态资源
- 乐观更新（操作立即反映在 UI，后台同步）
