## 1. 为什么要重做

### 1.1 现状问题

当前 PLM 文档管理是一个 **"能存文件的列表"**，而不是工程文档管理系统：

| 能力 | 当前状态 | 行业标杆 |
|------|---------|---------|
| 版本控制 | 只有 version 字段，手动填写 | 双层版本(Revision+Iteration)，自动递增 |
| 生命周期 | `transition_document` 未实现，状态不可流转 | 完整状态机 + 审批工作流 |
| 文件管理 | 单文件附件，无预览 | 多附件 + 在线预览 + Check-in/Check-out |
| 变更管理 | 无 | 已发布文档修改必须关联 ECN |
| 文档关系 | 仅 related_type/related_id | 引用、派生、替代、零件-文档 等多种关系 |
| 审批流程 | 无 | 多级审批 + 数字签名 + 审计追踪 |
| 模板 | 无 | 按类型预设模板，自动填充元数据 |
| 搜索 | 仅标题 | 全文搜索 + 元数据过滤 + 关联搜索 |

### 1.2 第一性原理

从**使用工程文档的人**的视角出发：

1. **文档是产品数据的一部分** — 不是独立的文件柜，是与零件、BOM、ECN 紧密关联的产品知识载体
2. **已发布的文档是不可变的** — 任何修改必须创建新版本，旧版本永久保留
3. **每次变更都要有理由** — 已发布文档的修订必须关联 ECN，保证可追溯
4. **审批即承诺** — 审批记录等同数字签名，不可篡改、不可否认
5. **用户只关心最新有效版** — 但需要随时回溯任何历史版本

## 3. 核心概念

### 3.1 双层版本模型 (Revision + Iteration)

借鉴 PTC Windchill 的经典模型，适配硬件产品开发场景：

```
                    ┌─── Iteration (工作迭代) ───┐
                    │  .1  .2  .3  .4  .5        │
                    │  ○───○───○───●───           │  ← 每次 Check-in 自动 +1
 Revision (正式版本) │                             │
     A ─────────────┘                             │
                                                  │
     B ────────────── .1  .2  ●                   │  ← 发布时 Revision +1，Iteration 重置
                                                  │
     C ────────────── .1  ○                       │
                    └─────────────────────────────┘

○ = Working iteration (可编辑)
● = Released iteration (不可变)
```

**版本号格式**: `A.3`（Revision A, Iteration 3）

**Revision 规则**:
- 首次创建: `A.1`
- Check-in: Iteration +1（`A.1` → `A.2` → `A.3`）
- 发布(Release): 当前 Iteration 冻结为已发布
- 修订(Revise): Revision +1，Iteration 重置为 1（`A.3` → `B.1`）
- 修订只能从已发布状态发起，且必须关联 ECN

**Revision 序列**:
- 开发阶段: `A, B, C, D, ...`（字母序列，跳过 I, O, Q, S, X, Z 以避免混淆）

### 3.2 生命周期状态

```
                    ┌──────── 驳回(reject) ────────┐
                    │                              │
                    ▼                              │
  ┌──────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ 草稿 │───▶│ 评审中   │───▶│ 已批准   │───▶│ 已发布   │
  │Draft │    │In Review │    │Approved  │    │Released  │
  └──────┘    └──────────┘    └──────────┘    └──────────┘
     ▲                                            │
     │              修订(revise, 需关联ECN)        │
     ├────────────────────────────────────────────┘
     │
     │         ┌──────────┐    ┌──────────┐
     │         │ 已替代   │    │ 已废弃   │
     │         │Superseded│    │Obsolete  │
     │         └──────────┘    └──────────┘
     │              ▲                ▲
     │              │                │
     └──── 新版本发布时自动设置 ──────┘
```

**状态说明**:

| 状态 | 可编辑 | 可删除 | 说明 |
|------|--------|--------|------|
| `draft` | 是 | 是 | 工作草稿，仅作者和协作者可见 |
| `in_review` | 否 | 否 | 评审中，内容冻结，评审人可批注 |
| `approved` | 否 | 否 | 评审通过，等待正式发布 |
| `released` | 否 | 否 | 已发布，正式有效版本，不可变 |
| `superseded` | 否 | 否 | 被新版本替代，仅供历史参考 |
| `obsolete` | 否 | 否 | 已废弃，不再有效 |

### 3.3 文档类型体系

按硬件产品开发生命周期组织：

| 类型代码 | 类型名称 | 典型场景 | 默认模板 |
|---------|---------|---------|---------|
| `SPEC` | 产品规格书 | 产品功能/性能/接口定义 | 规格书模板 |
| `DWG` | 工程图纸 | 2D/3D 图纸、装配图 | — (文件上传) |
| `SOP` | 作业指导书 | 生产工艺、装配流程 | SOP模板 |
| `TST` | 测试报告 | 可靠性/性能/合规测试结果 | 测试报告模板 |
| `QP` | 质量计划 | 检验标准、抽样方案 | 质量计划模板 |
| `BOM_DOC` | BOM文档 | BOM 变更说明、替代料清单 | — |
| `CERT` | 认证文档 | 认证证书、合规声明 | — (文件上传) |
| `DS` | 数据手册 | 元器件/物料 Datasheet | — (文件上传) |
| `MAN` | 用户手册 | 产品说明书、维护手册 | 手册模板 |
| `OTHER` | 其他 | 不属于以上类型的文档 | — |

### 5.2 操作栏 — 状态感知的智能按钮

按钮根据文档状态和用户权限动态显示：

| 状态 | 主按钮 | 次要按钮 | 禁用条件 |
|------|--------|---------|---------|
| `draft` (未Check-out) | Check-out | 提交评审, 编辑, 删除 | 他人已 Check-out |
| `draft` (已Check-out, 本人) | Check-in | 取消Check-out, 编辑 | — |
| `draft` (已Check-out, 他人) | — | — | 全部禁用，显示 "XXX 正在编辑" |
| `in_review` | — | 批准, 驳回 (评审人可见) | 非评审人 |
| `approved` | 发布 | — | — |
| `released` | 修订 | 废弃 | 修订需关联 ECN |

### 5.6 文档列表页 — 极致交互体验

#### 5.6.1 设计理念

对标 Linear（键盘优先、瞬间过滤）、Notion（多视图、分组折叠）、Google Drive（悬停操作、密度控制）、Figma（网格/列表切换），为工程师打造**高信息密度、低操作成本**的文档工作台。

**核心原则**:
1. **零等待** — 所有过滤和搜索都是客户端即时响应，不刷页面
2. **渐进式发现** — 默认展示最常用的信息，更多细节通过悬停/展开渐进呈现
3. **键盘优先** — 核心操作（搜索、筛选、导航）全部支持键盘快捷键
4. **上下文不中断** — 单击预览而非跳转，在当前页面完成 80% 的操作

### 6.2 关键命令详细设计

#### create_document_v2

```yaml
Input:
  title: string (required)
  doc_type: string (required)    # SPEC/DWG/SOP/TST/...
  product_id: string             # 关联产品
  project_id: string             # 关联项目
  part_id: string                # 关联零件
  template_id: string            # 基于模板创建
  description: string
  content: string                # Markdown 正文
  tags: string[]
  confidential: bool

Output:
  id: string
  code: string                   # DOC-00042
  version_label: string          # A.1
  status: string                 # draft
```

**逻辑**:
1. 自动生成 Code（DOC-XXXXX）
2. 初始版本: Revision=A, Iteration=1, VersionLabel="A.1"
3. 若指定 template_id，将模板内容填入 Content
4. 创建初始版本快照（PlmDocumentVersion）
5. 设置 AuthorID = 当前用户, IsLatest = true

#### checkin_document

```yaml
Input:
  document_id: string (required)
  change_summary: string (required)  # 本次修改说明
  content: string                     # 更新的正文（可选）
  title: string                       # 更新的标题（可选）

Output:
  id: string
  version_label: string              # A.3 (iteration+1)
  iteration: int
```

**逻辑**:
1. 验证当前用户已 Check-out 此文档
2. Iteration += 1
3. 更新 VersionLabel = Revision + "." + Iteration
4. 创建 PlmDocumentVersion 快照
5. 清除 CheckedOutBy / CheckedOutAt
6. 若有文件变更，更新 PlmDocumentFile

#### revise_document

```yaml
Input:
  document_id: string (required)
  ecn_id: string (required)          # 必须关联 ECN
  reason: string                     # 修订原因

Output:
  id: string                         # 新文档记录的 ID
  code: string                       # 同一 Code
  version_label: string              # B.1
  revision: string                   # B
```

**逻辑**:
1. 验证文档状态为 `released`
2. 验证 ECN 存在且状态为 `draft` 或 `in_progress`
3. 旧文档标记为 `superseded`
4. 创建新文档记录：同 Code，Revision +1（A→B），Iteration=1
5. 复制内容和附件到新版本
6. 设置 ECNID
7. 创建"supersedes"关系
8. 新文档状态为 `draft`

### 6.3 ECN 联动逻辑

```
ECN 创建
  └── 变更范围包含文档修订
       └── 自动/手动触发 revise_document
            └── 文档 ECNID 指向此 ECN
                 └── ECN 任务中包含"文档修订"类型任务

ECN 发布
  └── 检查所有关联文档是否已发布
       ├── 是 → ECN 可以发布
       └── 否 → 阻止 ECN 发布，提示未完成的文档
```
